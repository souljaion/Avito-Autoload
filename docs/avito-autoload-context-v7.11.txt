1. **Маленькими порциями.** Чинить по одной задаче, проверять руками, потом
   следующую. Урок 5 фиксов за день: пакетные изменения = непредсказуемые
   поломки. Stop conditions в каждом промте — обязательно.

2. **iframe приемлем для embedded edit form.** Form.html имеет 5+ page-level
   JS globals (goodsTypes, apparels, goodsSubtypes, saved, isNewProduct) +
   DOM ID конфликты + DOMContentLoaded handlers. Альтернатива — partial
   refactor — слишком рискованна. iframe + ?inline=1 + postMessage =
   стабильное решение.

3. **Backfill только после ack оператора.** SQL UPDATE на существующие данные
   = всегда DRY RUN сначала, показать что будет изменено, ждать
   подтверждения.

4. **Никогда не менять схему/семантику полей фида без проверки на одном аккаунте.**
   Урок 6279ff7 (v7.11): изменение `<Id>` с product.id на avito_id казалось
   безобидным, но Авито воспринял смену Id как исчезновение старых объявлений
   и снёс ~300 записей с 4 аккаунтов. Правильный подход: тестировать на одном
   аккаунте с минимальным количеством товаров, проверять отчёт Авито, только
   потом масштабировать.

Причина: маршрутизация между RU провайдером оператора и Timeweb Amsterdam
(72.56.123.13). Пинг 100% потерь даже с мобильника.
Решение: включён Cloudflare proxy (оранжевое облачко) на A-запись `autoload`
в dash.cloudflare.com. После — пинг 73ms, всё работает.

## 1. ЦЕЛЬ ПРОЕКТА И БИЗНЕС-КОНТЕКСТ

Avito Autoload — веб-приложение для управления товарными объявлениями на Avito
(крупнейший российский маркетплейс). Система позволяет:

- Импортировать товары с Avito или создавать вручную
- Управлять каталогом: категории, фото, описания, цены
- Генерировать XML-фиды в формате Avito Autoload v3
- Загружать фиды через Avito API (OAuth2)
- Мониторить отчёты Avito (applied/declined)
- Собирать аналитику (просмотры, контакты, избранное) с маркерами эффективности
- Планировать публикации по расписанию (inline и массовое)
- Работать с моделями и фотопаками для мультиаккаунтных схем
- Загружать фото из публичных папок Яндекс.Диска (products + photo_packs)
- Управлять библиотекой шаблонов описаний (standalone, decoupled от аккаунтов)
- Автоимпорт новых объявлений каждые 3 часа
- Синхронизировать фото из CRM (отдельная БД)
- Мягкое удаление товаров с автоматической отправкой Status=Removed в фид Авито
- Импорт avito_id из XML-фида Авито (feed_importer)
- Импорт Excel-экспорта Авито (brand/goods_type/photos)
- Умная кнопка удаления: 🗑 если товар валиден для фида, ⚠ + ссылка на Авито иначе

Бизнес-модель: ресейл одежды/обуви/аксессуаров через несколько аккаунтов Avito.
Один товар (модель) может публиковаться на разных аккаунтах с уникализацией фото.

## 5. СТАТУСЫ ТОВАРОВ

  imported  — импортирован с Авито (не показывается в /products)
  draft     — создан через систему, черновик
  scheduled — в очереди на публикацию
  active    — активен на Авито
  published — опубликован через автозагрузку
  paused    — приостановлен
  sold      — продан (legacy; sold_detection.py использует)
  removed   — мягко удалён + removed_at → 48ч в фиде → cleanup

XML-фид включает:
  - active + scheduled + imported (если is_ready_for_feed=True)
  - removed (< 48ч, с avito_id) — минимальный <Ad>


## 6. is_ready_for_feed — ВАЛИДАЦИЯ ДЛЯ ФИДА

Imported: avito_id + (product_images или image_url) + brand + goods_type.
Не-imported: title + description + price + category + goods_type + subcategory
  + goods_subtype (v7.8: only if requires_subtype()) + product_images.
Описание: из шаблона аккаунта если use_custom_description=False.
v7.6: description_template_id > use_custom_description > account_template.


## 7. XML-ФИДЫ

Формат: Avito Autoload XML v3. Генерация через lxml (feed_generator.py).
URL: /feeds/{feed_token}.xml (feed_token — UUID из accounts).
<Id>: feed_ad_id если заполнен, иначе str(product.id) (v7.11).
Removed товары (< 48ч): минимальный <Ad> с Id+AvitoId+Status=Removed.
  НО: удаление imported через фид ненадёжно (Авито часто игнорирует).
<Delivery>/<DeliverySubsidy>: убраны из фида (v7.11, невалидные значения).
ВАЖНО: в Avito-кабинете URL должен быть с правильным UUID feed_token.


## 8. БЕЗОПАСНОСТЬ

HTTP Basic Auth. Публичные пути: /health, /feeds/*.xml, /media/*, /static/*.
Rate limiting (slowapi): /api/analytics/fees 10/min, /products/{id}/avito-status 30/min.
POST /products/new: валидация account_id перед INSERT (400, не 500).

## 10. AVITO IMPORT — СИНХРОНИЗАЦИЯ С АВИТО

Сервис: app/services/avito_import.py
  Каждые 3ч: GET items → UPDATE/CREATE/mark_removed.
  Missing на Avito → status=removed + removed_at → 48ч cleanup.

Сервис: app/services/autoload_sync.py
  Pass 1: отчёты → imported. Pass 2: Items API → avito_id match.
  Pass 3: fuzzy title → CREATE imported + backfill.

Сервис: app/services/feed_importer.py
  XML feed → match by avito_id/title → CREATE imported.

Сервис: app/services/excel_importer.py
  xlsx → brand/goods_type/size/color/avito_id/image_url/photos.

Сервис: app/services/sync_from_avito_export.py (v7.11)
  xlsx-экспорт кабинета Авито → полная синхронизация products для аккаунта.
  Create/Update/Delete. Проставляет feed_ad_id из "Уникальный идентификатор".
  UI: /admin/avito-sync (preview + apply).
